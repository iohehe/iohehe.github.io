0x01 Data Flow Analysis

数据流分析，描述的是How Data Flows on CFG(数据在控制流上的传递情况)。我们关注的是application-specific Data 不同场景的数据流分析需要关注的数据抽象(data abstraction)不同，其安全策略也不同(flow safe-approximation strategies)， 那么根据安全策略，也会制定相应的转化函数(transfer functions)和边(control-flow handlings)。

May analysis: outputs information that may be true (over-approximation). 可能的都报出来

Must analysis: outputs information that must be true (under-approximation). 报出来的必须对

两者应用于不同场景，都是为了结果是正确的。以结果的正确为目的，称之为safe-approximation。

0x02 Prelimiaries of Data Flow Analysis

Input and Output States

控制流图上无非有三种情形： 顺序、分支、汇聚。 要在控制流分析数据，需要了然数据在控制流上的传递情况。那么数据如何变化的呢。

一条statement, 执行前的状态是IN[s1]， 执行后的状态是 OUT[s1]。顺序执行，s2的input就是s1的output。

分流、汇聚依然。注意汇聚点有一个union符号^。 这样每一句statement执行前就是input state 执行后就相当于new了一个 output state。而每一个state都与一个program point相关，这里的program point通常指的就是这条statement的上一条(before)和下一条(after)语句。

根据program point的概念， 可以做如下定义：

In each data-flow analysis application, we associate with every program point a data-flow value that represents an abstraction of the set of all possible program states that can be observed for that point.

就是说对每一条statement的前后program point ，会关联一个数据流的值，这个值代表program states 在这一点的所有抽象，即在这一program point上所有可能的出现状态。

Control Flow’s Constraints

知道Basic Block的概念， 那么Control Flow 就分BB内，和BB外。

BB内的话没有分支， 那么就是状态顺序转化。

BB块的话，进入状态是s1的状态， 出状态就是sn的状态。

这里在定义正向，反向的块间控制流， 有些抽象。正反各有两条，第一条是定义出口， 正向为例。OUT[B]是出口， fb这个从右往左，依次连接基本块内每一条语句的转换函数。得到出口处的转化规则fb，以IN[B]为参数即可。而第二条则是定义入口， 入口处分支汇合的情况，因此我们需要^上这些OUT[P]。逆向反着来，同样的原理。

如图， 假设在d 这个 program point处有一个变量v的定义操作，那么在p到q这条执行路径上，没有对变量v做重新定义。

Reaching Definitions 是一个may analysis, 定义上没有说must, 那就是over-approximation
